## 1. 核心对比

### 1.1 创建型模式对比

| 场景 | 推荐模式 | 原因 |
|------|---------|------|
| 唯一实例 | 单例 | 天然唯一 |
| 创建一族对象 | 抽象工厂 | 一致性 |
| 创建单个对象，但有变体 | 工厂方法 | 可扩展 |
| 复杂对象，多步骤构造 | 建造者 | 分步构建 |
| 克隆而非 new | 原型 | 避免重复初始化 |

### 1.2 结构型模式对比

| 场景 | 推荐模式 | 原因 |
|------|---------|------|
| 接口转换 | 适配器 | 兼容已有类 |
| 增加功能 | 装饰器 | 不改原类 |
| 控制访问 | 代理 | 延迟、缓存、保护 |
| 简化复杂系统 | 外观 | 高层入口 |
| 抽象与实现分离 | 桥接 | 多维变化 |
| 树形结构 | 组合 | 统一 leaf/composite |
| 大量相似对象 | 享元 | 节省内存 |

### 1.3 行为型模式对比

| 场景 | 推荐模式 | 原因 |
|------|---------|------|
| 算法可替换 | 策略 | 客户端选择 |
| 算法骨架固定 | 模板方法 | 父类定义骨架 |
| 一对多通知 | 观察者 | 解耦 |
| 顺序访问 | 迭代器 | 统一遍历 |
| 多处理器串联 | 责任链 | 沿链传递 |
| 请求对象化 | 命令 | 撤销、队列 |
| 状态决定行为 | 状态 | 状态自动转换 |
| 状态快照 | 备忘录 | 撤销/事务回滚 |
| 集中通信 | 中介者 | 解耦同事类 |
| 数据结构稳定，操作多变 | 访问者 | 易于扩展操作 |
| 自定义 DSL | 解释器 | 解释执行 |

## 2. 容易混淆的对比

### 2.1 装饰器 vs 代理 vs 适配器

| 维度 | 装饰器 | 代理 | 适配器 |
|------|--------|------|--------|
| 目的 | 增加功能 | 控制访问 | 接口转换 |
| 接口 | 不变 | 不变 | 可能变 |
| 创建时机 | 运行时动态 | 通常提前 | 已有类补救 |

### 2.2 策略 vs 状态

| 维度 | 策略 | 状态 |
|------|------|------|
| 切换方 | 客户端 | 状态自动 |
| 关系 | 独立 | 互相知道 |

### 2.3 模板方法 vs 策略

| 维度 | 模板方法 | 策略 |
|------|---------|------|
| 实现 | 继承 | 组合 |
| 灵活性 | 较低 | 较高 |

## 3. 选型原则

### 3.1 何时**不该**用设计模式

- **简单场景**：只有 1-2 个类，不需要模式
- **不确定未来**：YAGNI（You Aren't Gonna Need It）
- **增加理解成本**：团队不熟悉反而难维护

### 3.2 何时**应该**用

- **有明确痛点**：重复代码、紧耦合、难扩展
- **团队熟悉**：团队能理解模式
- **行业惯例**：Spring AOP 用代理、Java I/O 用装饰器

### 3.3 实战选择流程

```
需求 → 识别问题
  ├─ 对象创建复杂？ → 创建型（工厂/建造者）
  ├─ 类结构不合理？ → 结构型（适配器/装饰器）
  └─ 对象交互复杂？ → 行为型（策略/观察者）
```

## 4. dify / ruoyi 仓库源码解读

### 4.1 dify 的模式选择

| 场景 | dify 的选择 |
|------|------------|
| 模型提供商（30+） | 策略模式（model_manager） |
| 工具创建 | 工厂方法（tool_manager） |
| 工作流节点 | 组合 + 模板方法 + 访问者 |
| 异步任务 | 命令模式（Celery） |
| 配置缓存 | 单例 + 享元 |
| 认证 | 装饰器（@login_required） |

### 4.2 ruoyi 的模式选择

| 场景 | ruoyi 的选择 |
|------|------------|
| 数据库适配 | 适配器（方言工厂） |
| 事务管理 | 装饰器（@Transactional） |
| Mapper 接口 | 代理（MyBatis 动态代理） |
| 业务外观 | 外观（ServiceImpl） |
| 事件通知 | 观察者（ApplicationEvent） |
| 数据权限 | 代理（AOP） |

## 5. 反模式：过度设计

```python
# ❌ 过度设计：1 个 if/else 用 5 个模式
class AnimalFactoryBuilder:
    def build_dog_factory(self): return DogFactory()
    def build_cat_factory(self): return CatFactory()
    # 实际只有一个 Animal 类

# ✅ 简单足够：直接写
class Animal:
    pass
```

**原则**：简单优于复杂，模式是工具不是目标。

## 6. 关键要点总结

- 模式不是越多越好，按需选择
- 创建型关注对象创建、结构型关注类结构、行为型关注对象交互
- 团队熟悉度 + 实际痛点 = 选型依据
- dify 偏策略 + 组合；ruoyi 偏适配 + 装饰 + 代理
- 简单优于复杂
