## 目标

搞清：**登录成功后凭证是什么、放哪、后续请求如何恢复 current user**。

### 学完应能讲清

1. 登录请求打到哪个 path、哪个文件  
2. 校验账号密码大致在 service 哪一层  
3. 成功后 token（或 cookie）如何回到浏览器（看 Set-Cookie / 响应体，以你抓包为准）  
4. 下一个需登录接口如何带凭证；服务端哪里读取  
5. 错误密码时的 HTTP 状态 / 业务错误（实测一次）

## 上手操作

### 1. 抓一条真实登录

1. 浏览器打开控制台登录页，打开 Network，勾选 Preserve log  
2. 用账号密码登录  
3. 找到登录相关请求（常见 path 含 `login`，完整前缀多为 `/console/api/...`）  
4. 记录：

```text
请求方法 / 完整 URL：
请求头里是否有 Cookie / Authorization：
响应状态码：
响应头 Set-Cookie 有哪些名字（可打码值）：
响应 JSON 里是否有 token 字段（有则记字段名，不记完整密钥）：
```

### 2. 源码跟读顺序（当前仓库锚点）

| 序 | 文件 | 关注点 |
|----|------|--------|
| 1 | `controllers/console/auth/login.py` | `@console_ns.route("/login")`、调用 `AccountService.login`、写 cookie 的 helper |
| 2 | `services/account_service.py` | `login` / 校验与发 token 相关逻辑（先搜 `def login`） |
| 3 | `libs/token.py`（或 login 文件 import 的 token 工具） | `set_access_token_to_cookie` / refresh / csrf |
| 4 | `extensions/ext_login.py` | flask-login 如何挂上 |
| 5 | `libs/login` 或 `controllers/console/wraps.py` | `login_required` / `with_current_user` 如何拒绝未登录 |
| 6 | `models/account.py` | Account 模型长什么样（扫一眼字段即可） |

在 `api/`：

```bash
rg -n "set_access_token_to_cookie|AccountService.login|login_required" \
  controllers/console/auth/login.py services/account_service.py libs extensions/ext_login.py | head -40
```

### 3. 故意失败一次

- 错误密码登录：看状态码与响应 body（记结论，不必背文案）  
- 登录成功后，复制一条 **需登录** 的 GET（Network 里右键 Copy as cURL）  
  - 去掉 Cookie / Authorization 再发一次 → 预期 401 或未授权类错误  

```bash
# 示意：把 <url> 换成你的接口；不要把真实 token 贴进 git 笔记
curl -s -o /dev/null -w "%{http_code}\n" "<需登录的完整 url>"
# 再带上浏览器里的 Cookie 头试一次对比
```

## 期望时序（跟读提纲）

```text
1. POST /console/api/login 进入 login 视图
2. 解析 body（email/password 等）→ 可能有限流/冻结检查
3. AccountService 校验账号 → 失败则业务错误 / 401 类
4. 成功：AccountService.login 生成 token 对（access/refresh 等，以代码为准）
5. 响应：set_*_cookie 写入浏览器；可能同时返回 JSON
6. 后续请求：浏览器自动带 Cookie（或前端带 Authorization）
7. ext_login / login_required / with_current_user 解析凭证 → current user
8. 视图业务逻辑继续
```

**对照概念文档：** Session 方案 vs JWT 方案——以你在 Dify 源码里看到的为准写一句「我们实际是哪种/混合」。

## 常见卡点

| 现象 | 说明 |
|------|------|
| 只在前端 localStorage 找 token | Console 可能主要靠 **HttpOnly Cookie**；以 Set-Cookie 与 `libs/token` 为准 |
| 登录 200 但后续仍 401 | Cookie 域/路径/Secure/SameSite；或 CSRF；看第二次请求是否带上 Cookie |
| 和 service_api 的 API Key 混淆 | 控制台用户登录 ≠ 应用 API Key（`apikey` / service_api） |
| 陷入加密解密细节 | 密码哈希算法名知道即可，2.2 重点是「身份如何贯穿请求」 |
| 改密码后旧 token | 可记一笔「是否有失效机制」，深挖可后置 |
