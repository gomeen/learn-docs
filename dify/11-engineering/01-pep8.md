## 1. 核心概念

### 1.1 PEP 8 是什么

PEP 8（Python Enhancement Proposal 8）是 Python 官方代码风格指南，
由 Guido van Rossum、Barry Warsaw 和 Nick Coghlan 在 2001 年首次发布，
现由 python.org 维护。它是 Python 社区的**事实标准**——不是法律，
但绝大多数开源项目都遵循它，目的是让所有人的代码读起来像同一个人写的。

PEP 8 的核心目标是**提高可读性**，所有规则都服从这条主线。

### 1.2 关键规则速览

PEP 8 覆盖了 10 多个维度，最重要的几条：

| 维度 | 规则 | 例子 |
|---|---|---|
| 缩进 | 每级 4 空格，不用 Tab | `def f():` 后缩进 4 空格 |
| 行宽 | 上限 79 字符（注释/文档字符串 72） | 项目可放宽到 100~120 |
| 空行 | 顶层函数之间 2 空行，类内方法 1 空行 | `def a():` 与 `def b():` 之间 2 行 |
| 引号 | 单引号与双引号等价，保持一致 | 一行内嵌套时双引号包单引号 |
| 命名 | 函数/变量 `snake_case`，类 `PascalCase`，常量 `UPPER_CASE` | `user_id`、`UserProfile`、`MAX_SIZE` |
| 导入 | 每行一个，分组（标准库 / 第三方 / 本地），组间空行 | `import os` 后空行，再 `import requests` |
| 比较 | 单行用 `is None` / `is not None`，布尔值不要比 `==` | `if x is None:` |
| 表达式 | 复杂条件分行，行尾用 `and` / `or` | 见下文示例 |
| 注释 | 行内注释与代码至少 2 空格分隔 | `x = x + 1  # 补偿偏移` |
| 文档字符串 | 所有公共模块、函数、类、方法应有 docstring | 三引号开头，PEP 257 |

### 1.3 为什么 dify 收紧到 120 字符

PEP 8 默认 79 字符太短，现代宽屏显示器浪费大量水平空间。
dify 在 `api/AGENTS.md` 中明确写到：

> Keep each line under 120 characters (including spaces).

这是一个常见的折中：
- 79 太窄，IDE 需要频繁换行
- 120 在 24~27 寸宽屏上可以分两栏阅读
- 在 13~14 寸笔记本上单栏刚好

### 1.4 PEP 8 不是教条

PEP 8 是指南不是法律。三种合法打破规则的情况：

1. **遵循规则会让代码可读性变差**（罕见，但要能解释为什么）
2. **与周围代码风格不一致**（为了风格统一，可临时违反）
3. **代码早于 PEP 8 存在**（不必重写历史）

一致性比"正确"更重要——同一份文件内风格不统一比偶尔超 80 字符更糟糕。

## 2. 代码示例

### 2.1 缩进、命名与导入顺序

```python
# 文件：example_pep8.py
"""演示 PEP 8 的核心约束。"""

import json
import os
import sys
from collections import defaultdict
from typing import Any

import requests

from extensions.ext_database import db


MAX_RETRY_COUNT = 3
DEFAULT_TIMEOUT_SECONDS = 30


def fetch_user_profile(user_id: str, include_email: bool = False) -> dict[str, Any]:
    """获取用户档案。

    Args:
        user_id: 用户唯一标识。
        include_email: 是否返回邮箱字段，默认 False（避免 PII 泄漏）。

    Returns:
        解析后的用户档案字典。
    """
    response = requests.get(
        f"https://api.example.com/users/{user_id}",
        params={"include_email": include_email},
        timeout=DEFAULT_TIMEOUT_SECONDS,
    )
    response.raise_for_status()
    return response.json()


class UserService:
    """用户服务：编排 profile、session 与 audit log。"""

    def __init__(self, db_session):
        self._db = db_session
        self._cache: dict[str, dict] = {}

    def get_cached_profile(self, user_id: str) -> dict | None:
        """如果缓存命中则返回 profile，否则返回 None。"""
        return self._cache.get(user_id)
```

**说明**：
- **导入分组**：标准库 → 第三方 → 本地，组间空行
- **命名**：常量 `MAX_RETRY_COUNT`、`UserService`、`fetch_user_profile`
- **方法之间**：类内方法之间 1 空行
- **类型注解**：参数与返回值都有标注，函数签名一眼可读
- **文档字符串**：公共函数和类都有 docstring，说明参数与返回值

### 2.2 常见违反 PEP 8 的反例

```python
# ❌ 错误示范 1：行宽超限
def calculate_total_amount(user_orders, discount_rate, tax_rate, shipping_fee, currency_code, include_gift_wrap):
    return sum(order.amount for order in user_orders) * (1 - discount_rate) * (1 + tax_rate) + shipping_fee

# ✅ 正确做法：在第一个参数处换行，或拆函数
def calculate_total_amount(
    user_orders: list,
    discount_rate: float,
    tax_rate: float,
    shipping_fee: float,
    currency_code: str,
    include_gift_wrap: bool = False,
) -> float:
    ...
```

```python
# ❌ 错误示范 2：用 Tab 缩进
def bad():
	return "Tab indent"  # IDE 会立即报混合缩进错误

# ✅ 正确做法：4 个空格
def good():
    return "Four-space indent"
```

```python
# ❌ 错误示范 3：比较 None 用 ==
if user is None == False:  # 双重否定 + == 比较 None
    ...

# ✅ 正确做法：is not None
if user is not None:
    ...
```

### 3.1 后端 PEP 8 配置：行宽 + 工具链

## Coding Style

**解读**：
- 第 51 行：dify 选 **Ruff** 而非 flake8/black 组合。Ruff 是 Rust 编写的 linter，速度比 Python 工具链快 10~100 倍，且兼容 PEP 8 全部规则

- `.ruff.toml` 是 Ruff 的配置文件，可以覆盖默认规则、添加项目专属规则（如 dify 的命名约定）

### 3.2 命名与类型系统约束

**文件位置**：`/Users/xu/code/github/dify/api/AGENTS.md`
**核心代码**（行 56-67）：

```markdown
### Naming Conventions

- Use `snake_case` for variables and functions.
- Use `PascalCase` for classes.
- Use `UPPER_CASE` for constants.

### Typing & Class Layout

- Code should usually include type annotations that match the repo's current Python version (avoid untyped public APIs and "mystery" values).
- Prefer modern typing forms (e.g. `list[str]`, `dict[str, int]`) and avoid `Any` unless there's a strong reason.
- For dictionary-like data with known keys and value types, prefer `TypedDict` over `dict[...]` or `Mapping[...]`.
```

**解读**：
- 第 57-59 行：dify 严格遵循 PEP 8 的命名约定
- 第 63 行：要求**所有公共 API 必须有类型注解**——这是 PEP 8 没强制但 dify 强制要求的
- 第 65 行：使用现代 typing 语法（`list[str]` 而非 `List[str]`），需要 Python 3.9+
- 第 66 行：**TypedDict 优先于 `dict[...]`** —— 这是 dify 的"风格收紧"，目的是让 dict 的 key/value 类型显式可见

## 4. 关键要点总结

- PEP 8 的核心目标是**可读性**，不是教条
- 关键约束：4 空格缩进、行宽（dify 120）、2 空行分隔、导入分组、`snake_case`/`PascalCase`/`UPPER_CASE`
- **dify 的风格收紧**：120 字符行宽、Ruff 工具链、TypedDict 替代 `dict`、公共 API 必须有类型注解
- PEP 8 之外用工具自动检查（`ruff check`、`ruff format`），不要靠人眼
- 一致性 > 正确：偶尔违反 PEP 8 比风格不统一好
