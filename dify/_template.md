## 🧪 章节小验证（checkpoint）模板

> 粒度：按阶段 README 的「模块」分组（约 3～6 篇知识点一组），每组 1 篇验证。  
> 命名：与知识点统一序号，`NN-*-<slug>.md`（如 `07-*-python-basics.md`），插在对应文档组之后。  
> 学习节奏：**读完一组 → 做对应 `NN-*-…` 小验证 → 下一组**。

```markdown
# 小验证：{小组名}

> 覆盖：
> - [文档 A](./xx.md)
> - [文档 B](./yy.md)
>
> 预计：30～90 分钟 · 本地练习或改 dify 仓库

## 背景
{1 段：为什么练这个、和 dify 的关系}

## 需求
{可验收的小改动，具体可执行；禁止空泛「请总结 XXX」}

## 提示
{可选路径/入口，不给完整答案}
- 路径示例：`/Users/xu/code/github/dify/api/...`

## 验收标准
- [ ] ...
- [ ] ...

## 延伸（选做）
{可选加深项}
```

**验证类型约定**：
- **语言基础**（`01-fundamentals` 前半）：本地小脚本 / 小模块即可，不强制改 monorepo
- **框架 / 项目相关**（`02-backend` 及之后，含 `*-in-dify`）：优先「在 `/Users/xu/code/github/dify` 改一个小功能点」风格的需求

## 🔒 必须遵守的规则

1. **篇幅与节奏**：
   - 单篇以「快扫能懂」为目标：核心概念 + 可运行示例 + 要点
   - **不要**在单篇里写大段仓库源码解读、练习题清单、参考资料堆砌
   - 动手验证统一放到阶段 **章节小验证（`NN-*-主题.md`）**

2. **代码规范**：
   - 示例代码：10-50 行，优先完整可运行
   - 若需点名 dify 位置：在「关键要点」加 **1 行路径锚点** 即可，不贴大段源码、不写行号精读
   - **基础入门篇**允许教学向精简，并注明完整形态见专题文档

3. **每个知识点至少包含**：
   - 清晰的学习目标与前置知识
   - 1 个独立可运行示例（或等价最小片段）
   - 关键要点总结（可含 1 行项目路径锚点）
   - **不要求**每篇附练习题；练习在对应 checkpoint 完成

4. **链接策略**：
   - 文档间跳转：用相对路径 `../folder/file.md`
   - dify 仓库文件：用绝对路径 `/Users/xu/code/github/dify/api/xxx`
   - **不要**使用 `https://github.com/...` 这种链接（官方文档 URL 除外）

5. **Sighting（露面）原则**（详见 [`_common/SIGHTING.md`](../_common/SIGHTING.md)）：
   - 正文若出现**其他专题文档才正式讲授**的概念，必须在首次显著出现处加 `详见 [标题](相对路径)`
   - **学习目标 / 关键要点 / checkpoint 需求** 不得要求掌握后置专题；越权练习改为「学完 XXX 后再做」或移入归属文档
   - 同一文档内同一主题只链一次，避免刷屏
   - 相对路径从当前文件计算
