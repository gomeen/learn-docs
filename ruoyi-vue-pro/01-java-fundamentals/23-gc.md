## 1. 核心概念

### 1.1 如何判定"对象已死"？

**方法 1：引用计数法**
- 每个对象有引用计数，被引用一次 +1，引用失效 -1，归 0 时死亡
- 缺点：无法处理循环引用（`A a = new A(); A b = new A(); a.field = b; b.field = a;`）

**方法 2：可达性分析法**（Java / HotSpot 用这个）
- 从 **GC Roots** 开始搜索，被引用到的对象为"存活"，其余为"死"
- GC Roots 包括：
  - 栈帧局部变量表
  - 静态变量引用的对象
  - 方法区常量引用的对象
  - JNI 句柄

### 1.2 GC 四大经典算法

| 算法               | 过程                                    | 优点       | 缺点       |
|------------------|---------------------------------------|----------|----------|
| **标记-清除**       | 先标记死对象，再清除                            | 简单       | 内存碎片     |
| **复制**           | 把存活对象复制到另一块区域，整片清理                  | 无碎片      | 浪费一半空间   |
| **标记-整理**       | 先标记死对象，再把存活对象移动到一端                  | 无碎片      | 移动开销大   |
| **分代**           | 新生代用"复制"，老年代用"标记-整理"                | 折中方案     | 复杂       |

### 1.3 分代理论

观察到的事实：**大多数对象朝生夕死**。基于此：

```
新生代 (Young)                   老年代 (Old)
+---+---+---+---+------+        +----------+
| E |S0|S1|                    |          |
+---+---+---+                  |          |
                ↑ 熬过 15 次 GC |          |
                +--------------+----------+
```

- 大多数对象在 **Eden** 创建，**Minor GC** 时死亡
- 仍存活的对象复制到 Survivor（S0 / S1 互相交替）
- 熬过 15 次 GC 后晋升到 **Old**
- 老年代满时触发 **Full GC**（应用会停顿很久——Stop-The-World）

### 1.4 常见 GC 实现

| 实现                | 算法             | 适用         |
|-------------------|----------------|------------|
| Serial GC        | 复制/标记-整理       | 单线程、客户端     |
| Parallel GC      | 多线程复制          | 吞吐量优先      |
| **G1 GC**        | 分区收集           | 大堆、低延迟     |
| ZGC             | 并发标记 / 着色指针     | 4MB 以下的停顿    |

### 1.5 Stop-The-World（STW）

GC 时必须暂停所有 Java 线程（修改对象引用），叫 Stop-The-World。G1 / ZGC 目标是**减小 STW 暂停时间**。

## 2. 代码示例

### 2.1 引用计数法的循环引用问题

```java
// 文件：RefCycleDemo.java
public class RefCycleDemo {

    static class Node {
        Node next;
    }

    // Python 等语言会立即内存泄漏（引用计数）
    // Java 不受影响（可达性分析）
    public static void main(String[] args) {
        Node a = new Node();
        Node b = new Node();
        a.next = b;
        b.next = a;
        a = null;       // 引用断开
        b = null;       // GC 时可达性分析发现是不可达的
    }
}
```

### 2.2 `System.gc()` 慎用

```java
// 文件：SystemGcDemo.java
public class SystemGcDemo {
    public static void main(String[] args) {
        new Object();   // 没有引用，很快死亡
        System.gc();    // 请求 GC（不保证立刻执行）
    }
}
```

`System.gc()` 只是"提示"，不保证立即触发；且会触发 Full GC。在生产中**应避免**。

### 2.3 手动触发 GC 日志观察

```bash
java -Xmx256m -Xlog:gc* -jar app.jar
```

GC 日志示例：
```
[2.345s][info][gc] GC(0) Pause Young (Normal) 24M->8M(256M) 5.123ms
[5.234s][info][gc] GC(1) Pause Young (Normal) 32M->16M(256M) 8.456ms
```

解读：
- `24M->8M`：GC 前 24MB，GC 后 8MB，腾出了 16MB
- `(256M)`：堆总容量
- `5.123ms`：STW 停顿

## 3. 关键要点总结

- 判断对象存活：**引用计数**（简单但有循环引用）和**可达性分析**（Java 用这个）
- 三大算法：标记-清除 / 复制 / 标记-整理
- 分代模型：新生代（复制算法）+ 老年代（标记-整理）
- 业务系统首选 **G1 GC**，关注 STW 暂停时间
