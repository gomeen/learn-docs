# 后端「会 / 不会」自检清单

> 面向：有前端经验、正在补后端。  
> 用途：逐项对照 → 只标 **会** 或 **不会** → 再据此生成知识地图与 2 周计划。

---

## 怎么用（必读）

### 只有两种状态

| 标记 | 含义 |
|------|------|
| `[ ]` **不会** | 默认。任一「会的判定」做不到、答不全、要搜答案才算数 → 保持不会 |
| `[x]` **会** | **该条下全部「会的判定」都能独立完成**（不查资料、不看提示） |

规则：

1. **不允许中间态**。觉得「大概知道」= **不会**。
2. **判定是清单，不是感觉**。每条下面有可勾的子项；子项有一条做不到 → 整条仍是 **不会**。
3. **先闭卷自测**。查文档之后再改标记会高估自己。
4. **口述类**：对着空气说满要求时长/要点；卡壳超过 10 秒或漏关键点 → 不会。
5. **动手类**：必须真的做过（本机/项目），不能只是「觉得能做」。

### 0.1 HTTP 与 API 契约

#### [ ] P0-01 五种 HTTP 方法语义与幂等

**会的判定（须全部满足）：**

- [ ] 不查资料写出：GET / POST / PUT / PATCH / DELETE 各自「对资源做什么」各一句话。
- [ ] 正确回答：下列哪些在 HTTP 语义上视为幂等？`GET, POST, PUT, PATCH, DELETE, HEAD`（答案要点：GET/PUT/DELETE/HEAD 幂等；POST 不幂等；PATCH 语义上通常不保证幂等）。
- [ ] 举出一个业务例子说明：**POST 创建订单连点两次**可能产生两笔订单；并说出至少一种后端防重手段名称（如：幂等键 / 唯一约束 / token 防重，任一种并说清放在哪一层即可）。
- [ ] 说明：为什么「更新用户全部资料」更常映射 PUT，「只改昵称」更常映射 PATCH（各一句）。

**不会的典型表现：** 只会说「GET 查 POST 改」；说不清幂等；认为 PATCH 一定幂等。

**文档：** [`_common/14-api-protocols/01-http-protocol.md`](./_common/14-api-protocols/01-http-protocol.md)、[`02-rest-api-design.md`](./_common/14-api-protocols/02-rest-api-design.md)

#### [ ] P0-02 状态码：401 / 403 / 400 / 404 / 500 与前端处理

**会的判定：**

- [ ] 用业务场景各举一例（不许重复场景）：
  - 返回 **401** 的情况
  - 返回 **403** 的情况
  - 返回 **400**（或 422）的情况
  - 返回 **404** 的情况
  - 返回 **500** 的情况
- [ ] 明确区分：**401 = 未证明身份（或凭证无效）**；**403 = 已证明身份但无权限**。不能混用。
- [ ] 说出前端收到 401 时至少一种合理动作（如：清登录态并跳转登录 / 尝试 refresh 再重试，按你们栈说一种并自洽）。
- [ ] 说出：业务错误（如「余额不足」）是否必须用 4xx 而不是 200+业务码——能说出**一种**常见约定及它的利弊各一句（不要求唯一真理，但要自洽）。

**不会的典型表现：** 「401 和 403 差不多」；所有错误都返回 200；说不清前端分支。

### 0.2 认证：你是谁

#### [ ] P0-06 Cookie + Session 完整链路

**会的判定：**

- [ ] 按时间顺序口述完整链路（不得跳步），至少包含：
  1. 用户提交账号密码  
  2. 服务端校验通过后创建 session（内存/Redis/DB 任说一种存放处）  
  3. 响应 `Set-Cookie` 写入 session id（名称可举例 `sessionid` / `sid`）  
  4. 浏览器后续请求自动带上 `Cookie`  
  5. 服务端用 session id 取出用户身份  
- [ ] 回答：session 数据主要存在**服务端**还是浏览器？（服务端）
- [ ] 回答：用户点「退出」时，服务端至少应做哪一件事？（销毁/失效 session；可选同时让 cookie 过期）
- [ ] 回答：多台后端机器时，session 存在单机内存会有什么问题？一种解决方向是什么？（会话粘滞 / 集中存储 Redis 等，任一种）

**不会的典型表现：** 以为 session 内容全存在 cookie 里；说不清退出时服务端要做什么。

**文档：** [`_common/07-authentication/02-session-cookie.md`](./_common/07-authentication/02-session-cookie.md)

#### [ ] P0-07 JWT 结构、签名能力边界与存放

**会的判定：**

- [ ] 说出 JWT 三段名称：**Header / Payload / Signature**（或头、载荷、签名）。
- [ ] 正确判断正误并改错：
  - 「JWT 签名能防止别人**读取** payload」→ 必须判为**错**，并说明 payload 默认只是编码、可被解码查看。  
  - 「JWT 签名能防止别人**篡改** payload 且不被发现」→ 必须判为**对**（在密钥未泄露前提下）。
- [ ] 举出 payload 里常见的至少 2 个声明用途（如 `sub` 用户 id、`exp` 过期时间、`iat` 签发时间）。
- [ ] 对比 `localStorage` 存 access token 与 `HttpOnly Cookie` 存 token：各写 **1 个优点 + 1 个风险**（XSS 读存储 vs CSRF 与 cookie 自动携带等，说对方向即可）。
- [ ] 回答：用户退出登录时，若 JWT 无服务端黑名单，已发出的 token 在过期前是否仍可能被接受？→ **是**；并说出一种缓解（短过期 + refresh / 黑名单 / 版本号，任一种）。

**不会的典型表现：** 以为 JWT 内容加密不可见；以为退出后 JWT 一定立刻失效。

#### [ ] P0-08 Access Token + Refresh Token

**会的判定：**

### 0.3 授权：你能干什么

#### [ ] P0-10 认证 vs 授权；401 vs 403

**会的判定：**

- [ ] 各用一句话定义：
  - 认证（Authentication）：确认你是谁  
  - 授权（Authorization）：确认你被允许做什么  
- [ ] 给定场景，强制选 401 或 403（不可「都行」）：
  1. 未带 token 访问需登录接口 → **401**  
  2. 带了合法 token，但角色是访客却调删除用户接口 → **403**  
  3. token 签名错误或过期 → **401**（按常见实践）  
- [ ] 回答：前端菜单根据角色隐藏了「删除」按钮后，后端删除接口还要不要鉴权？→ **要**；漏了会发生什么？（直接调 API 越权成功）

**不会的典型表现：** 用 401 表示没权限；认为前端隐藏等于安全。

**文档：** [`_common/08-authorization/README.md`](./_common/08-authorization/README.md)

## P1 · 数据怎么存、怎么查对、怎么不乱

### 1.1 SQL 与建模

#### [ ] P1-01 手写基础 CRUD SQL

**会的判定（须能手写，允许表名自拟但语法正确）：**

- [ ] `SELECT` 指定列 + `WHERE` 一个条件 + `ORDER BY` + `LIMIT`（或等价分页）。
- [ ] `INSERT` 插入一行并列出列名。
- [ ] `UPDATE ... SET ... WHERE`（**必须**带 WHERE；能说明漏 WHERE 的后果：全表更新）。

- [ ] 说明 `NULL` 与空字符串不是一回事；`WHERE col = NULL` 为何写不对（应 `IS NULL`）。

**不会的典型表现：** 只会 GUI 点点点；UPDATE/DELETE 习惯性不写 WHERE。

#### [ ] P1-02 JOIN：INNER 与 LEFT

**会的判定：**

- [ ] 用「用户表 + 订单表」说明：
  - INNER JOIN：只保留两边都匹配的行  
  - LEFT JOIN：保留左表全部，右表无匹配则为 NULL  
- [ ] 手写一版（可简化）：查询所有用户及其订单数——若用户无订单仍要出现，应选哪种 JOIN？→ **LEFT**；并说明聚合时可能用到 `COUNT` + `GROUP BY` 的方向。
- [ ] 指出：JOIN 条件写在 `ON` 上；把过滤条件误塞进 `ON` 与写在 `WHERE` 对 LEFT JOIN 结果可能不同（至少知道「有影响、要小心」）。

**不会的典型表现：** 只会一个表查询；分不清 INNER/LEFT 结果集差异。

**文档：** [`02-sql-join.md`](./_common/21-sql/02-sql-join.md)

#### [ ] P1-03 主键、唯一约束、外键在防什么

**会的判定：**

- [ ] 主键：表内唯一标识一行；一张表通常一个主键；不能为 NULL（按常见关系库实践）。
- [ ] 唯一约束：防业务重复（如同一 email 只能注册一次）；与主键区别：主键还承担行标识/常作外键目标。
- [ ] 外键（概念）：子表引用父表合法 id，防「订单指向不存在的用户」；即使项目不用 DB 外键，也要在应用层保证等价约束——能复述这句意思。
- [ ] 举一例：没有唯一约束时，并发双击注册同一邮箱可能怎样？（两条重复用户）

**不会的典型表现：** 觉得「程序里判断一下就行」且不知并发下会双插。

### 1.3 事务

#### [ ] P1-09 事务：要么都成功要么都失败

**会的判定：**

- [ ] 场景：创建订单行 + 扣减库存，要求原子性。无事务且第二步失败时，数据会出现什么业务后果？（有订单无扣库存或相反，说清一种真实事故）

#### [ ] P2-03 穿透 / 击穿 / 雪崩（定义 + 各一种应对）

**会的判定：**

- [ ] **穿透**：查不存在的数据，缓存不命中且每次打到 DB。应对至少 1：缓存空值 / 布隆过滤。

**不会的典型表现：** 三个名词背混；只知道「加缓存」不知道过期风暴。

- [ ] 当接口 500 时，知道去应用日志找堆栈/错误消息；能说明至少要收集：时间、路径、request_id（若有）、用户、入参摘要。
- [ ] 实际完成过一次：本地或项目环境用 curl/httpie 打通一个需登录接口。**没做过 = 不会。**

**不会的典型表现：** 只会点页面；500 时只说「后端挂了」无下一步。

**文档：** [`13-linux-shell`](./_common/13-linux-shell/README.md)、[`04-network-commands.md`](./_common/13-linux-shell/04-network-commands.md)

## P5 · 异步与消息（可第二阶段）

#### [ ] P5-01 同步 API vs 后台任务

**会的判定：**

- [ ] 列出至少 3 类应异步的工作：发邮件/短信、大报表、视频转码、AI 长推理、批量导入（任 3）。
- [ ] 说明同步硬做长任务的问题：超时、占 worker、用户一直转圈。
- [ ] 描述一种异步交互：接口先返回任务 id → 客户端轮询或 WebSocket/SSE 拿结果。

**不会的典型表现：** 所有逻辑都塞在一个 HTTP 请求里跑数分钟。

**文档：** 项目队列文档；[`_common/02-mq/`](./_common/02-mq/README.md)

#### [ ] P5-02 消息队列解决什么

**会的判定：**

- [ ] 说出至少 3 个动机：削峰、解耦、异步化、最终一致性（任 3）。
- [ ] 明确：MQ **不是**「带队列的 HTTP」；发布方与消费方生命周期可分离。
- [ ] 举一例：下单后发通知——下单服务与通知服务通过消息解耦的好处（通知挂了不一定挡住下单，视设计）。

**不会的典型表现：** 觉得有了 REST 就不需要队列思想。

**文档：** [`01-concepts.md`](./_common/02-mq/01-concepts.md)

---

#### [ ] P5-03 至少一次投递与消费幂等

**会的判定：**

- [ ] 说明「至少一次」：消息可能被重复投递；消费者可能执行多次。
- [ ] 因此消费者必须幂等：同一消息 id 处理多次结果与一次相同。
- [ ] 给出至少 1 种幂等手段：唯一键去重表、业务状态机、SETNX 记处理过的 msgId。
- [ ] 说明：若消费者「先干活再 ack」与「先 ack 再干活」失败时的不同风险方向（重复执行 vs 丢消息）。

**不会的典型表现：** 假设每条消息只会被处理恰好一次。

**文档：** [`05-reliability.md`](./_common/02-mq/05-reliability.md)

---

#### [ ] P5-04 分布式锁使用时机（概念）

**会的判定：**

- [ ] 场景：多实例部署下，定时任务可能同时跑两次——需要互斥时考虑分布式锁。
- [ ] 说明：单机 `synchronized`/进程内锁**不能**跨进程/跨机器互斥。
- [ ] 说出至少一种实现载体：Redis / ZooKeeper / etcd（知一名即可）。
- [ ] 知道锁要设过期防死锁；业务仍尽量做幂等（锁不是万能）。

**不会的典型表现：** 多实例下仍只用本地锁；认为有锁就不必幂等。

**文档：** [`_common/04-distributed-locks/`](./_common/04-distributed-locks/README.md)

## 前端背景 · 常见误解（会 = 能指出错在哪并改述正确版）

每条：**先判断对错 → 用正确表述重说一遍**。做不到 = 不会。

#### [ ] M-01

命题：「前端校验过了，后端可以不再校验。」  
**会的判定：** 判错；正确表述：安全与正确性以后端为准，前端只做体验与减少无效请求。

#### [ ] M-02

命题：「页面慢一定是前端渲染问题。」  
**会的判定：** 判错；能说出至少拆成：DNS/TCP、TTFB、接口总时间、SQL/下游、下载与渲染；并知道先看 Network 时间线。

#### [ ] M-03

命题：「用了 ORM 就不用懂 SQL。」  
**会的判定：** 判错；能举 N+1、慢 SQL、事务边界、索引至少 2 个仍依赖 SQL 认知的点。

#### [ ] M-04

命题：「Token 放 localStorage 最方便，所以是最佳实践。」  
**会的判定：** 判错或「不绝对」；能说出 XSS 窃取风险，并对比 HttpOnly Cookie 的取舍（CSRF 等）。

#### [ ] M-05

命题：「401 和 403 差不多，前端统一踢去登录页就行。」  
**会的判定：** 判错；401 未认证/凭证问题，403 已认证无权限；403 不应一律当未登录处理。

#### [ ] M-06

命题：「加上 Redis 一定会更快更稳。」  
**会的判定：** 判错；能说出不一致、穿透/击穿/雪崩、多一层故障点。

#### [ ] M-07

命题：「CORS 报错是服务器防火墙拦了，和浏览器无关。」  
**会的判定：** 判错；CORS 由浏览器执行同源策略；服务器通过响应头声明是否放行；curl 常能通。

#### [ ] M-08

命题：「学后端必须先上微服务和 K8s。」  
**会的判定：** 判错；应先掌握单体下的请求链路、数据、鉴权、部署基础。

## 综合情景题（会 = 按「标准答案要点」覆盖，闭卷）

每题下方是**必须覆盖的要点**；缺任一组要点 → 本情景 = 不会。建议录音或默写后对照。

#### [ ] S-01 登录态全链路

**题目：** 用户登录成功后，下一次请求后端如何识别身份？凭证存在哪？过期与登出怎么做？

**会的判定（须覆盖）：**

- [ ] 指出一种完整方案（Session **或** JWT，选你主栈）并讲完：登录签发 → 客户端携带方式 → 服务端校验 → 得到 userId。  
- [ ] 明确凭证/会话的存储位置（服务端 session 存储 / 客户端 token 存储位置）。  
- [ ] 过期：如何过期（TTL/exp）；过期后客户端表现。  
- [ ] 登出：服务端做什么、客户端做什么；若 JWT 无黑名单，承认「过期前仍可能有效」并给缓解。

---

#### [ ] S-02 水平越权下单/看单

**题目：** 用户 A 把订单 id 改成 B 的，请求会怎样？后端在哪拦？

**会的判定：**

- [ ] 现象：若只校验登录，A 可能读到/操作 B 的订单。  
- [ ] 拦截点：业务层/查询条件带 owner，或统一的数据权限组件。  
- [ ] 返回：403 或 404（防枚举）的一种合理选择及理由一句。  
- [ ] 强调：前端路由与按钮隐藏无效。

---

#### [ ] S-03 列表接口 3 秒慢

**题目：** 管理端表格 3 秒才出数，按什么顺序排查？

**会的判定（须有顺序意识，至少 5 步级）：**

- [ ] 前端确认：是等待接口慢还是渲染慢（Network 看 TTFB/总时间）。  
- [ ] 后端确认：接口总耗时、是否多次下游调用。  
- [ ] DB：慢 SQL、是否全表扫、是否缺索引、`EXPLAIN`。  
- [ ] ORM：是否 N+1。  
- [ ] 其他：锁等待、缓存是否命中、序列化大字段、分页是否失效一次拉全表。  
- [ ] 给出你个人固定的排查顺序清单（写下来），逻辑自洽即可。

---

#### [ ] S-04 创建订单连点两次

**题目：** 如何避免两笔订单？幂等键放哪？

**会的判定：**

- [ ] 问题本质：POST 非幂等 + 重复提交/重试。  
- [ ] 至少 2 层防护中的 2 个：前端防连点（不足够单独依赖）、幂等键（Idempotency-Key）、DB 唯一约束（业务单号）、token 防重。  
- [ ] 说明服务端如何用幂等键：先查「此键是否已处理」再创建。  
- [ ] 说明仅靠前端 disable 按钮为什么不够。

---

#### [ ] S-05 改密码后旧 JWT

**题目：** 用户改密码后，旧 JWT 是否仍可用？产品期望与实现？

**会的判定：**

- [ ] 默认纯 JWT：在 exp 前**仍可用**（无额外机制时）。  
- [ ] 产品常期望：改密后全局下线。  
- [ ] 至少 2 种实现方向：token 版本号/密码时间戳写入 payload 并与 DB 比对、黑名单、全员踢下线清 session、短 access + 吊销 refresh。

---

#### [ ] S-06 更新后仍见旧数据

**题目：** 更新 DB 后页面仍旧数据，可能路径与修复？

**会的判定：**

- [ ] 至少列出 3 个可能层：浏览器缓存、CDN/HTTP 缓存、Redis/应用缓存、只读副本延迟、前端本地状态未刷新。  
- [ ] 对应修复方向至少 2 个：写后删缓存、Cache-Control、刷新列表、读主库。  
- [ ] 能描述你如何**验证**是哪一层（换无痕、看响应头、查 Redis key、看是否命中副本）。

---

#### [ ] S-07 本地 CORS 联调

**题目：** `localhost:3000` → `localhost:8080` 报 CORS，查谁、改什么？

**会的判定：**

- [ ] 判定是浏览器同源策略；不是「端口不通」时的唯一解释。  
- [ ] 优先检查后端/网关 CORS 配置：允许的 Origin、Methods、Headers、是否 `credentials`。  
- [ ] 说明预检 OPTIONS 失败也会表现为 CORS 错误。  
- [ ] 说明用 curl 打 8080 可能成功——不能用来证明浏览器端 CORS 已好。  
- [ ] 给出一种本地可行配置思路（允许 `http://localhost:3000`，不要用含 credentials 时的 `*`）。

---

#### [ ] S-08 只有「500 服务器错了」

**题目：** 还要哪些信息才能继续查？

**会的判定（须列出至少 6 项信息中的 6 项）：**

- [ ] 发生时间（含时区）  
- [ ] 请求方法与完整路径  
- [ ] 环境（dev/staging/prod）与版本/提交号  
- [ ] 用户/租户标识  
- [ ] request_id / trace_id  
- [ ] 请求入参（脱敏后）  
- [ ] 响应体错误码/消息  
- [ ] 服务端堆栈或错误日志原文  
- [ ] 是否可稳定复现、单用户还是全站  

缺收集意识、只会说「看一下日志」而无字段清单 → 不会。
